Design principles

01
Calm by default

Color and weight are reserved for what matters — the meal, the vote, the threshold. Everything else recedes. The app should feel like a well-set table, not a notification feed.

02
One accent, used deliberately

The warm orange carries every primary action. It appears once per screen — on the button the user is here to press. Never use it decoratively.

03
Thumb-first layouts

Primary actions live in the bottom half of the screen. Nothing critical lives in the top corners. The tab bar is within reach at all times.

04
Platform-agnostic rendering

Every effect must render identically on iOS and Android. No translucency, no blur, no platform-specific widgets. If it needs a conditional, it probably doesn't belong.

05
Soft depth, never heavy

Elevation reads through low-alpha layered shadows and hairline borders — not hard outlines or colored fills. Cards should feel lifted, not boxed in.

06
Thin screens, shared components

Screen files are routing + state only. Every visual element lives in lib/widgets/. If you're writing a Container with decoration in a screen file, extract it first.

Foundations · 02

Brand

MenuApp is a meal-planning tool for couples and small families. The brand is warm, approachable, and focused — like a good cookbook. The orange accent signals appetite and energy without feeling urgent. Voice is plain, warm, and direct.

Three words: plain, warm, encouraging.

Reference · 16

Do & Don't

Do
  • All colors from AppColors. Zero hard-coded hex anywhere in widget files.
  • All radii from AppRadii. All shadows from AppShadows.
  • Use AppButton, AppCard, AppTextField — not Material equivalents.
  • Skeleton for list loading; spinner only for blocking actions.
  • Pair every status color with a dot or icon — never color alone.
  • Keep screens thin: BlocBuilder + layout only. Logic in Cubit.
  • Use accent-deep (#C2570A) for accent text on light surfaces.
  • Respect safe areas with SafeArea on every screen root.
Don't
  • No ElevatedButton, FloatingActionButton, AppBar, BottomNavigationBar from Material.
  • No inline BoxDecoration in screen files — extract to a named widget.
  • No blur, no translucency, no BackdropFilter.
  • No single-offset or colored drop shadows — always 2-layer, low alpha.
  • No tinted pill with same-hue text — neutral surface + dot badge only.
  • No Colors.grey, Colors.orange, or any un-tokenized color.
  • No radial gauges — use linear meters for all progress.
  • Don't put business logic or HTTP calls in widget files.

Appendix · B

Build-agent handoff brief

Read this if you are a coding agent turning this document into Flutter. It tells you exactly what to build, where to put it, and how "done" is judged.

Goal

Produce a custom Flutter component library that renders the specimens above pixel-for-token. Every widget reads its color, radius, shadow, typography, and motion from the central theme — no inline literals anywhere in widget files. No screens are in scope for this task; build the kit, then compose screens from it.

File structure

lib/ theme/ app_colors.dart // §15 — AppColors class app_radii.dart // §15 — AppRadii class app_shadows.dart // §15 — e0/e1/e2/e3 BoxShadow lists app_typography.dart // §05 — AppTextStyles, AppTypography.mono app_theme.dart // §15 — buildTheme() wiring everything together widgets/ buttons/ app_button.dart // primary / secondary / ghost / destructive inputs/ app_text_field.dart app_toggle.dart app_filter_chip.dart app_segmented.dart app_stepper.dart cards/ app_card.dart // base card with e1 + hairline + lg radius group_card.dart // avatar + name + threshold + chevron recipe_card.dart // name + cuisine + tags + status badge badges/ app_badge.dart // dot + label, enum for ok/warn/danger/info/muted app_tag.dart // outline chip, dietary tags lists/ app_list_row.dart // avatar + title + subtitle + trailing progress/ vote_meter.dart // linear bar + threshold marker + vote count nav/ app_nav_bar.dart // pill bottom nav, 2–3 tabs app_page_header.dart // large-title collapsing header feedback/ app_snackbar.dart // dark + light variants app_dialog.dart // title + body + cancel/confirm app_sheet.dart // modal bottom sheet with grab handle states/ skeleton_list_item.dart empty_state.dart // icon + title + body + optional action button error_state.dart // inline error with retry gallery/ component_gallery.dart // scrollable screen showing every widget + all states

Rules of engagement

  • One source of truth: all values from theme/. Zero hard-coded hex, radius, or shadow in widget files.
  • Implement every state in §13 via WidgetStateProperty.resolveWith — no duplicate widgets for pressed/disabled states.
  • Opaque surfaces only — no blur, no translucency.
  • Elevation via 2-layer BoxShadow lists — never Material elevation int.
  • Every widget exposes a named, minimal API. No positional styling params.
  • Meet all §14 accessibility requirements: 48px tap targets, Semantics labels, text scaling.
  • Honor MediaQuery.disableAnimations in all animated widgets.
  • Add a gallery entry per component showing all states before marking done.
